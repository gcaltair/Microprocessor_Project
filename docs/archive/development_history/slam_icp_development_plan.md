# SLAM 建图与 ICP 匹配开发计划

## 1. 目标与约束

依据课程需求文档，本阶段必须交付的核心能力包括：

- 基于轮编码器的实时里程计
- 2D Occupancy Grid Map (OGM) 增量建图
- 轻量级 scan matching，用于修正累计里程计漂移，`ICP` 可作为直接实现方案
- 基于 OGM 的自主探索、Start -> Exit 导航、Exit -> Start 回程
- 全部运行在 `STM32F446RE + FreeRTOS` 上，不能依赖上位机在线计算

量化 benchmark 只看两件事：

- `Start -> Exit`，上限 `120s`
- `Exit -> Start`，上限 `120s`

因此开发策略必须是：

1. 先做出稳定的“可跑通闭环”
2. 再做地图质量和定位精度收敛
3. 最后围绕 benchmark 时间做性能优化

## 2. 当前代码基线判断

从当前仓库看，已经具备以下基础：

- `FreeRTOS` 主干任务已建立：`ControlTask`、`LiDARParseTask`、`CommTask`、`SafetyTask`
- `LiDAR DMA -> scan buffer -> queue` 的数据链路已经存在
- 编码器测速与基础里程计已经存在
- 蓝牙命令、运行时状态输出、LiDAR 启停都已接入
- 当前工程可以成功编译

当前不足也很明确：

- 还没有真正接入 `MappingTask / LocalizationTask / PlanningTask / ExplorationTask`
- 仓库当前源码中看不到 `icp_map.c`，但旧构建产物里出现过该对象文件，说明代码基线存在“历史构建残留和源码状态不完全一致”的风险
- 里程计目前更接近“可用骨架”，尚不足以直接支撑可靠建图和回程
- LiDAR 数据目前主要用于打包传输，尚未进入地图更新和位姿修正闭环

结论：

- 你们现在不是从 0 开始
- 但也还没有真正进入“SLAM 已集成、只差调参”的阶段
- 最合理的路线是“冻结现有 RTOS 骨架，在其上补齐 SLAM 最小闭环”

## 3. 推荐总体方案

建议采用“最小可行 SLAM”路线，而不是一开始就做复杂融合。

### 3.1 建议架构

- `ControlTask`:
  10ms，负责 IMU/编码器更新、速度控制、基础里程计预测
- `LiDARParseTask`:
  接收并整理一帧完整激光扫描
- `MappingTask`:
  20Hz 左右，读取最新 scan 和当前位姿，更新 OGM
- `LocalizationTask`:
  10Hz 到 20Hz，使用“里程计预测 + ICP 修正”输出修正后位姿
- `PlanningTask`:
  事件驱动，基于 OGM 运行 A*
- `ExplorationTask`:
  2Hz 左右，前沿点检测，生成新的探索目标
- `SafetyTask`:
  维持心跳、故障状态、资源监控

### 3.2 地图与定位策略

- 地图表示：`int8_t` 或 `uint8_t` OGM，优先固定大小，避免动态内存
- 建议地图范围：`4.8m x 4.8m`
- 建议分辨率：`0.05m`
- 建议网格尺寸：`96 x 96`
- 内存预算：约 `9KB`，可控

### 3.3 ICP 实现策略

不要上来做重型 SLAM。建议做 MCU 可承受的轻量 ICP：

- 输入不是全量点云，而是降采样后的有效点
- 每帧保留 `60-120` 个点
- 使用里程计提供初值
- 每次 ICP 迭代 `5-8` 次即可
- 对匹配距离设置门限，超限直接丢弃
- 如果 fitness 差，直接回退到 odometry，避免错误校正污染地图

### 3.4 第一版不建议做的事

- 不建议一开始就上 EKF/图优化
- 不建议一开始就做全局回环检测
- 不建议把 scan matching、建图、规划都堆进同一个高优先级任务

第一版目标是：

- 能持续建图
- 能压住里程计漂移
- 能找到出口
- 能基于已建地图回到起点

## 4. 分阶段开发计划

## Phase 0: 基线冻结与接口清理

目标：确保后续算法开发不被工程问题反复打断。

任务：

- 清理“源码缺失但旧对象残留”的问题，确认 `icp_map` 是否需要恢复或重建
- 明确 `pose`、`scan`、`map`、`path` 的结构体定义，避免跨模块随意读写全局变量
- 统一锁顺序：`odom -> control -> pid` 或固定单向顺序
- 给关键共享状态补 `volatile` 或快照接口，避免任务并发读取出错
- 记录当前可编译基线，作为后续回退点

完成标准：

- 工程稳定编译
- 关键数据结构固定
- 共享资源访问规则文档化

预计时间：

- `1-2` 天

## Phase 1: 里程计与激光数据链路加固

目标：让 `scan + pose` 具备进入 SLAM 的基本可信度。

任务：

- 修正当前里程计积分逻辑，明确角度单位和更新方式
- 给每帧 LiDAR scan 打时间戳或序号，并记录对应 pose 快照
- 为 scan 添加基础滤波：
  - 去除无效距离
  - 去除极近/极远噪声
  - 限制点数上限
- 输出调试接口：
  - 当前 pose
  - 每帧点数
  - 里程计累计位移
  - 丢帧/溢出计数

完成标准：

- 小车原地旋转和直线运动时，pose 变化方向正确
- 连续多帧 scan 点数稳定
- LiDAR 与 odometry 可以一一对应

预计时间：

- `2-3` 天

## Phase 2: 最小 OGM 建图闭环

目标：不做 ICP，先用 odometry + LiDAR 建出可读地图。

任务：

- 实现 `OccupancyGrid_Init`
- 实现世界坐标到网格坐标转换
- 实现激光射线更新：
  - 终点占用
  - 沿途自由
- 限制地图边界，越界时安全丢弃
- 提供地图导出接口：
  - 蓝牙简化文本
  - 或离线导出给上位机可视化

完成标准：

- 在简单走廊或矩形障碍环境下，地图轮廓可辨认
- 墙体方向大致正确
- 未出现明显的数组越界和地图污染

预计时间：

- `3-4` 天

## Phase 3: 接入轻量 ICP 位姿修正

目标：把“会漂”的 odometry-only 地图，提升到“可回程”的地图。

任务：

- 建立 scan 点到局部地图或上一帧参考点集的匹配接口
- 实现轻量 `ICP_Match(scan, predicted_pose)`：
  - 输入：里程计预测位姿
  - 输出：修正后的位姿与 fitness
- 加入失败保护：
  - 匹配点过少不更新
  - fitness 超阈值不更新
  - 单帧修正幅度超阈值不更新
- 建图时改用“修正后位姿”，而不是纯里程计位姿

完成标准：

- 重复经过同一路段时，地图重影明显减少
- 直线墙面不再快速弯曲
- 回到已知区域时，位姿误差可收敛

建议指标：

- 局部重复定位误差 `< 10cm`
- 航向误差 `< 5 deg`

预计时间：

- `4-5` 天

## Phase 4: 探索、出口搜索与回程

目标：从“能建图”走到“能完成 benchmark”。

任务：

- 实现 frontier detection
- 对 frontier 或出口候选点生成全局目标
- 在 OGM 上实现 `A*`
- 输出离散路径点给控制层
- 加入局部避障策略

建议顺序：

1. 先做“已知目标点 A* 走通”
2. 再做“frontier 选点”
3. 最后接“出口发现 -> 切换目标 -> 返回起点”

完成标准：

- 已知目标下，路径规划可稳定输出
- 能覆盖未知区域并逐步扩展地图
- 找到出口后可切换到回程模式

预计时间：

- `4-6` 天

## Phase 5: 实时性、鲁棒性与 benchmark 收敛

目标：从“能完成”走到“跑得快、跑得稳”。

任务：

- 统计各任务周期、执行时间、堆栈余量、队列水位
- 优化 LiDAR 点数和 ICP 迭代次数
- 调整地图分辨率和规划触发频率
- 为以下故障定义 Fault 行为：
  - LiDAR 超时
  - 编码器异常
  - 规划失败
  - 队列阻塞
- 完成 `Start -> Exit` 与 `Exit -> Start` 的 benchmark 调参

优先优化顺序：

1. 保证控制任务不抖动
2. 再压低 scan matching 开销
3. 再缩短探索与规划路径

完成标准：

- 系统长时间运行无死锁
- benchmark 双程可以稳定完成
- 运行时间开始逼近评分区间

预计时间：

- `3-4` 天

## 5. 模块拆分建议

建议拆成以下文件级模块，避免逻辑继续堆在 `freertos.c`：

- `slam_types.h`
  - Pose、Scan、Grid、Path、Frontier 等基础结构
- `occupancy_grid.c/.h`
  - OGM 初始化、坐标变换、射线更新、占用概率更新
- `scan_preprocess.c/.h`
  - 点云筛选、降采样、极坐标转笛卡尔
- `icp.c/.h`
  - 最近邻、刚体变换估计、fitness 评估、失败回退
- `mapping_task.c/.h`
  - 地图更新流程
- `localization_task.c/.h`
  - odom 预测 + ICP 修正
- `planning.c/.h`
  - A*
- `exploration.c/.h`
  - frontier detection 与目标选择

这样做的好处：

- FreeRTOS 调度逻辑和算法逻辑解耦
- 便于单独测试 `ICP` 和 `OGM`
- 后面写文档和视频讲解时更清楚

## 6. 测试与验收计划

## 6.1 单元级

- `Odometry_Update()`：
  给定模拟左右轮速度，检查 `x/y/theta` 是否按预期变化
- `OccupancyGrid_Update()`：
  给定静态 scan，检查网格占用结果
- `ICP_Match()`：
  给定已知平移/旋转偏差的点集，检查能否收敛
- `A*`：
  给定固定障碍图，检查路径是否可达

## 6.2 集成级

- 静止环境连续扫描：
  地图不应持续漂移
- 直线往返：
  回到起点附近时误差应可接受
- 原地旋转：
  地图墙面不应明显扭曲
- 小型迷宫：
  能持续探索，未出现任务阻塞

## 6.3 Benchmark 级

- 记录每次：
  - 出口时间
  - 回程时间
  - 是否中途碰撞
  - 是否因定位失败卡死
- 每次 benchmark 后保留：
  - 地图结果
  - pose 轨迹
  - 关键参数

## 7. 任务优先级建议

如果多人并行开发，建议按下面拆分：

- 成员 A：
  `odometry + control + runtime stats`
- 成员 B：
  `LiDAR preprocess + OGM`
- 成员 C：
  `ICP + localization`
- 成员 D：
  `A* + frontier exploration + benchmark orchestration`

集成顺序必须固定：

1. A/B 先打通 `pose + map`
2. C 再接入 `ICP`
3. D 最后接规划和探索

否则会出现“规划、定位、地图同时不稳定，无法定位问题来源”的情况。

## 8. 主要风险与规避

### 风险 1: ICP 过重，抢占控制周期

规避：

- 强制限点数
- 限迭代次数
- fitness 不佳直接跳过
- ICP 只在新 scan 完成时运行

### 风险 2: 地图分辨率过高，内存和更新时间爆炸

规避：

- 第一版固定 `96 x 96 @ 5cm`
- 先保证可用，再决定是否细化

### 风险 3: 里程计方向、角度单位、坐标系不统一

规避：

- 在 `slam_types.h` 写清楚坐标定义
- 全项目统一使用弧度或角度，不能混用

### 风险 4: scan 与 pose 未对齐

规避：

- 每帧 scan 入队时保存 pose 快照
- 不要在异步任务里直接读取“当前全局 pose”作为历史 scan 的位姿

### 风险 5: 过早追求复杂算法

规避：

- 第一版只做最小可行闭环
- EKF、回环、复杂避障全部放到 benchmark 稳定之后

## 9. 建议里程碑

- `M1`:
  基线冻结，里程计和 LiDAR 数据链路稳定
- `M2`:
  odometry-only OGM 能出图
- `M3`:
  ICP 接入后地图明显更稳
- `M4`:
  A* 和 frontier 打通，能自主找到出口
- `M5`:
  完成 Exit -> Start 回程
- `M6`:
  双程 benchmark 开始进入稳定收敛阶段

## 11. 最终建议

对你们当前工程，最正确的顺序不是“先把所有高级算法都写出来”，而是：

1. 冻结 RTOS 基线
2. 修好 odometry 与 scan 对齐
3. 先做 odometry-only 建图
4. 再接轻量 ICP
5. 最后做探索、出口、回程和 benchmark 调优

如果顺序反过来，后面很容易陷入“地图漂、定位漂、规划也不稳，但不知道到底是谁的问题”。
