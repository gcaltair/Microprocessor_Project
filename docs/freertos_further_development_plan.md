# FreeRTOS 后续开发计划

## 文档状态

- 版本：`v1.2`
- 更新日期：`2026-05-12`
- 适用范围：当前 STM32F446 + FreeRTOS + LiDAR SLAM 固件

---

## 1. 当前系统状态

当前系统已经不是“RTOS 改造计划”阶段，而是“RTOS 上的功能系统集成阶段”。

现有能力包括：

- `FreeRTOS` 多任务调度
- 编码器 + IMU odometry
- LiDAR scan 解析
- 轻量 `LocalizationTask`
- 占据栅格建图
- 实时 ASCII 地图输出
- 已知目标格点导航初版

当前主任务链路：

1. `ControlTask`
2. `LiDARParseTask`
3. `LocalizationTask`
4. `MappingTask`
5. `CommTask`
6. `SafetyTask`
7. `NavigationTask`（逻辑服务型模块，已接入调度链）

---

## 2. 当前架构分层

### 2.1 状态层

系统当前已经实际形成三层位姿概念：

- `odom_pose`
  - 高频
  - 平滑
  - 控制距离闭环当前主要依赖它

- `control_pose`
  - 面向控制层的融合位姿
  - 当前仍保持保守使用

- `corrected_pose / EST`
  - 面向建图与导航
  - 来自 `LocalizationTask`

### 2.2 功能层

- **控制层**
  - 以稳定性优先
  - 当前不允许 LiDAR 修正直接破坏 PID 闭环

- **建图层**
  - 统一使用 `corrected_pose`

- **导航层**
  - 当前使用地图格点和估计位姿
  - 已具备最小路径执行能力

## 4. 当前进行中的阶段

### Phase 3B：控制可用的融合定位

当前状态：**进行中**

目标不是“把 LiDAR 直接喂给 PID”，而是：

- 保持控制稳定
- 让 LiDAR 修正逐步进入更高层状态估计
- 为导航提供更可信的位置参考

当前已知约束：

- LiDAR 修正有低频和抖动
- 不能直接进入角度闭环
- 控制距离闭环仍主要看 `odometry`
- 地板材质和正反向不对称会显著影响结果

当前结论：

**Phase 3B 仍需继续收口，但不再是阻止 Phase 4A 启动的绝对前置条件。**

### Phase 4A：已知目标点基础导航

当前状态：**已启动**

已经具备：

- 地图格点目标输入
- 基础路径规划接口
- 导航状态机
- 分段执行路径

当前命令：

- `Jx,y`：导航到目标格点
- `J`：查看导航状态
- `C`：取消导航

当前定位：

- 这是已知目标格点导航
- 不是完整自主探索
- 不是 return-to-start

## 5. 当前命令与调试接口

### 5.1 建图与地图查看

- `M / N`：开启/停止 LiDAR
- `G`：地图摘要
- `X1 / X2 / X4 ...`：导出 ASCII 地图
- `U1 / U0`：开启/关闭 `0.5 s` 实时缩略地图流

### 5.2 位姿与定位

- `O`：`ODOM / ENC / LMOVE / EST / CTRL / pred / corr / slow`
- `O` 同时包含 `MOVE cmd/target/progress/remain` 相对位移诊断信息，并保留最近一次动作结果直到下一次命令或复位
- `O` 同时包含 `LMOVE valid/cmd/progress/dl/dr`，直接显示最近一次完成的相对位移段轮程
- `O` 同时包含 `CTRLDBG / LCTRL / TCTRL`，用于 PID 调参和抖动复盘
- `O` 新增 `PIDDBG / LIMDBG / MOTORDBG / ENCDBG`，用于速度环、角度环、位置环联合调参：
  - `PIDDBG`：各 PID 环的 setpoint、current、error、P/I/D 分量、输出、积分量和输出饱和状态
  - `LIMDBG`：位置环 base speed 原始值/限幅值、终端限速、角度 turn 原始值/限幅值和死区激活状态
  - `MOTORDBG`：PID PWM、加死区后的实际 PWM、左右电机方向
  - `ENCDBG`：编码器脉冲增量、原始轮速、滤波轮速和 odometry 缩放后轮速
- `P`：定位状态摘要

### 5.2.1 PID 调参

- `PK`：查看所有 PID 环参数
- `PKA / PKL / PKR / PKP`：查看角度、左轮速度、右轮速度、位置环参数
- `PK<loop>,kp,ki,kd`：临时设置 RAM 中的 PID 参数，例如 `PKA,0.05,0,0.001`
- 参数不写 Flash，控制器重启后恢复 `PID_system_init()` 中的默认值
- `WSleft,right`：速度环隔离测试命令，直接设置左右轮速度目标值，单位 `m/s`，绕过位置环和角度环，只保留速度 PID 到电机输出。例如 `WS-0.03,-0.03`
- `WS` 命令当前限幅为左右轮目标速度绝对值不超过 `0.12 m/s`；使用 `S` 退出并停车
- 上位机 PID 面板和 CLI 只作为调参辅助，不能替代急停和人工安全观察

### 5.3 编码器标定

- `K`：查看当前正反向标定系数
- `Klf,lr,rf,rr`：修改标定系数
- `R0`：复位编码器计数、odometry 位姿、LocalizationTask 和导航状态
- `D<actual_distance_m>`：优先基于最近一次完成的 `P{x},{y}` 相对位移段轮程；若没有可用段，则回退到当前累计轮程，再结合实测位移给出新的标定建议

### 5.4 导航

- `Jx,y`：导航到地图格点
- `J`：查看导航状态
- `C`：取消导航

## 6. 当前主要问题

### 6.1 运动尺度误差

当前实测已经说明：

- 前进和后退存在不对称
- 地板材质变化会带来明显影响
- 单次往返后实际偏移仍可达 `10 ~ 15 cm`
- 当前里程计在部分工况下仍可能出现 `10% ~ 20%` 的尺度误差

最近一轮固件修正已经完成：

- odometry 位移累计直接使用原始编码器速度，避免平滑速度造成累计缩放
- 平移积分使用中点航向
- 相对位移控制按命令前进轴投影计算进度，避免浅弧行驶时把实际路程低估成起点直线位移
- 基础运动控制闭环统一使用 odometry 参考，不再把 `control_pose` 慢速修正混入 PID 主链路
- `L / R` 蓝牙手动转向命令已切回相对转向语义，和 `A+angle` 保持一致
- 串口调试补齐累计轮程、原始计数、复位和标定建议接口
- `D...` 标定建议改为优先使用最近一次完成的相对位移段，避免前序摆位或上一段动作污染当前标定
- 针对 `P1,0` 终点附近抖动，位置环最后 `0.15 m` 取消强制最低速度，并增加终端速度上限，从 `0.08 m/s` 收缩到约 `0.015 m/s`
- 倒车相对移动补齐 `-MAX_BASE_SPEED` 对称限幅，避免 `P-1,0` 反向速度输出过大时放大终点抖动
- 移动中的角度修正按当前 `base_speed` 收缩限幅，避免终点附近一侧轮子被角度环拉成反转
- 角度控制加入 `1 deg / 2 deg` 迟滞，降低转角目标附近的反复启停
- `O` 输出新增 `CTRLDBG pos_err/ang_err/turn/l_sp/r_sp/pwm`，用于下一轮实车定位抖动来源
- `O` 输出新增 `LCTRL`，保留最近一次相对位移动作结束前最后一轮驱动控制输出，避免动作结束后只能看到停止态
- `O` 输出新增 `TCTRL`，保留最近一次原地转向有输出时的控制状态，避免转角结束后只能看到静止态
- `PK` / `PKA` / `PKL` / `PKR` / `PKP` 可查看 PID 参数，`PK<loop>,kp,ki,kd` 可临时设置 RAM 中的 PID 参数，重启后恢复编译默认值
- 上位机新增 PID 面板和 `car-pid-tuner` CLI，用于可视化 `CTRLDBG / LCTRL / TCTRL` 并为后续 AI agent 调参保留 JSONL 数据入口
- `rollback/before-deadzone-debug` 分支地面三环调参结果：速度环默认值更新为左右轮 `Kp=3500 Ki=6000 Kd=0`，角度环默认值更新为 `Kp=0.008 Ki=0 Kd=0.0005`，位置环暂保持 `Kp=0.8 Ki=0 Kd=0`。不采用 `Kp=4500 Ki=9000`，因为 `WS/P0.50` 样本出现低速编码器反向尖峰后 PID 反向大输出。后续应优先修速度反馈抗尖峰和到达阈值，而不是继续加大位置环
- 编码器速度反馈增加抗尖峰策略：降低速度滤波权重、收紧最大合理轮速，并拒绝最多两个单周期大幅反向 raw speed 采样，避免低速量化/干扰直接让速度 PID 反向大输出
- 短距离 `P0.3,0` 复测显示抗尖峰后速度反馈稳定，但终点仍约少走 `5 cm`；位置到达阈值已从 `0.05 m` 收紧到 `0.025 m`，用于减少短距离相对位移提前结束
- `P0.3,0` / `P0.5,0` 在 `0.025 m` 到达阈值下已稳定完成；为降低直行终点约 `2-3 deg` 航向残差，移动中角度修正比例从 `0.80` 调整为 `1.00`，仍保留随 `base_speed` 收缩的限幅以避免轮速反向

首轮硬件量距已经暴露出更具体的问题：

- 前进 `1.0 m` 时，`ODOM` 约 `0.984 m`、`MOVE progress` 约 `0.957 m`，实体仅约 `0.82 m`
- 前进阶段 `ENC dl/dr = 0.793 / 1.176 m`，左右轮比例明显不对称
- 首轮倒车记录里的 `ENC dl/dr = -0.337 / 0.430 m` 属于累计量；若与前进累计值做差，倒车单段更接近 `dl/dr = -1.130 / -0.746 m`
- 新版 `LMOVE / CAL diag` 隔离测试进一步确认：前进实体 `0.83 m` 时 `LMOVE dl/dr = 0.838 / 1.069 m`，建议 `lf=0.990`、`rf=0.776`
- 新版 `LMOVE / CAL diag` 隔离测试进一步确认：倒车实体 `-1.12 m` 时 `LMOVE dl/dr = -1.048 / -0.857 m`，建议 `lr=1.069`、`rr=1.308`
- 当前合并建议命令：`K0.990,1.069,0.776,1.308`
- 应用合并系数后，前进实体距离改善到 `0.97 m`，但单次 `CAL diag` 仍有 `left_err=-9.0%`、`right_err=5.2%`
- 应用合并系数后，倒车本轮实体距离为 `-0.85 m`，与上一轮 `-1.12 m` 差异较大，说明单次 1 m 标定结果存在明显波动
- 重新整理/固定线路后，默认 `K1.000,1.000,1.000,1.000` 下前进实体为 `0.88 m`、倒车实体为 `-0.96 m`，左右不对称显著降低
- 当前判断更新：线束接触、编码器信号干扰、供电/地线或线束运动是大幅波动的重要嫌疑，应先处理硬件稳定性再继续软件标定
- 重新理线后，`K0.957,0.992,0.896,1.030` 完成连续 `3` 次前后 `1 m` 往返测试，最终与起点误差约 `5 cm`
- `L / R` 相对转向语义已恢复；转向过程卡顿和可能超调已加入第一轮抖动缓解修改，仍需实车复测

因此当前问题已经不再只是“泛泛的 10% ~ 20% 误差”，而是：

- 前进方向以轮比例和左右不对称为主
- 倒车方向当前同样以反向左右轮不对称和比例偏差为主
- 单次标定建议容易过拟合，后续应固定同一组 `K` 做多轮重复并按中位数小步调整
- 理线后默认系数表现明显改善，下一步应先用默认 `K` 建立新的重复性基线
- 当前推荐固件默认编码器系数为 `K0.957,0.992,0.896,1.030`
- 转向环品质也会影响后续标定重复性
- 终点附近抖动当前重点看 `CTRLDBG`：如果 `l_sp/r_sp` 仍频繁一正一负，应继续收紧移动中转向限幅；如果 PWM 正负翻转但速度目标稳定，应检查速度环和电机死区补偿

因此当前不应以“精准回零”作为导航前提。

### 6.2 建图精度仍有限

当前地图已经“可读”，但仍受以下因素影响：

- 运动速度过快
- 转弯速度过快
- odometry 尺度误差
- scan matching 仅为轻量实现

### 6.3 内存风险

当前资源占用大致为：

- RAM：`98.85%`
- FLASH：`22.06%`

这意味着后续新增模块必须优先考虑 RAM。

## 7. 后续开发优先级

### Priority 1：收口 Phase 4A

下一步优先把基础导航做成“可重复验证”的最小闭环：

- 路径可视化
- 到达容差
- 失败状态输出
- 重规划入口
- 更清晰的导航调试信息

### Priority 2：收口 Phase 3B

继续推进但保持保守：

- 不把 `corrected_pose` 直接送入 PID
- 继续维护分层位姿
- 逐步评估 `control_pose` 是否还有必要继续保留增强
- 优先完成“单段隔离”的 `R0 -> P{x},{y} -> O -> D... -> K...` 人工标定闭环，并把结果写入任务记录
- 优先复测 `R0 -> P1,0 -> O`、`R0 -> P-1,0 -> O`、`R0 -> A90 -> O`、`R0 -> A-90 -> O`，确认终点与转向抖动是否下降

### Priority 3：资源优化

重点检查：

- 地图规划缓存
- 导航路径缓存
- 运行时临时数组

目标是为后续 frontier / return-to-start 释放内存余量。

### Priority 4：进入完整导航状态机

在基础导航稳定后，再进入：

- frontier exploration
- exit detection
- return-to-start

---

## 8. 推荐开发顺序

推荐按以下顺序推进：

1. 稳定当前基础导航
2. 补导航路径显示
3. 加失败恢复和重规划
4. 做 RAM 收口
5. 再考虑 frontier
6. 最后做回程状态机

---

## 9. 当前阶段判断

当前系统已经具备“继续做导航”的条件，但前提是：

- 导航必须按“容错系统”来设计
- 不把 dead reckoning 当作高精度真值
- 允许分段执行、重定位、再规划

因此后续计划不需要推翻，但需要继续坚持：

- 分层
- 小步快跑
- 每一步都可测
- 遇到问题可快速回退
