# SLAM / ICP 开发进度状态

## 1. 当前结论

截至 2026-05-12，项目已经从“仅有 FreeRTOS 控制骨架”推进到“可运行的建图链路 + 初版已知目标导航”阶段。

当前最准确的状态是：

- `Phase 0 ~ Phase 2`：完成
- `Phase 3A`：完成
- `Phase 3B`：部分完成，仍在调参和验证
- `Phase 4A`：已启动，已有第一版最小可用实现

一句话概括：

**系统已经具备稳定控制、LiDAR 建图、轻量定位修正、实时地图输出，以及面向已知目标格点的基础导航能力；但运动尺度、建图精度和导航健壮性仍不足以视为完整自主导航系统。最新一轮里程计调试已经通过 `LMOVE / CAL diag` 获得可回写的左右轮正反向标定建议，下一步是应用新系数并复测。**

## 2. 已完成内容

### 2.1 FreeRTOS 主链路

系统当前已经具备以下主要任务链路：

- `ControlTask`
- `LiDARParseTask`
- `LocalizationTask`
- `MappingTask`
- `CommTask`
- `SafetyTask`

这些任务已经形成稳定的数据流：

`odometry / IMU -> pose snapshot -> LiDAR scan -> LocalizationTask -> corrected_pose -> MappingTask`

### 2.2 占据栅格地图

当前地图参数为：

- 固定尺寸：`96 x 96` cells
- 分辨率：`0.05 m / cell`
- 实际覆盖：`4.8 m x 4.8 m`
- 地图原点：`(-2.4 m, -2.4 m)`

地图已经支持：

- `G`：地图摘要
- `X1 / X2 / X4 ...`：ASCII 地图导出
- `R`：机器人当前位置标记
- LiDAR 工作期间直接查看 `X1 / X2`
- `U1`：每 `0.5 s` 输出一次整张缩略地图

### 2.3 轻量定位修正

`LocalizationTask` 已经接入，并维护以下位姿：

- `predicted_pose`
- `corrected_pose`
- `EST`
- `CTRL`

当前策略是：

- 建图统一使用 `corrected_pose`
- 导航/调试使用 `EST`
- 控制主链路仍以 `odometry` 为主
- `CTRL` 只保留为控制融合位姿实验入口，不再直接替换 PID 输入

### 2.4 调试与标定接口

当前可用命令包括：

- `M / N`：启动/停止 LiDAR 建图
- `O`：输出 `ODOM / ENC / EST / CTRL / pred / corr / slow`
- `P`：输出定位/建图状态摘要
- `G`：输出地图摘要
- `X1 / X2 / X4 ...`：导出不同缩略等级 ASCII 地图
- `U1 / U0`：启停实时缩略地图流
- `K`：查看编码器正反向标定系数
- `Klf,lr,rf,rr`：设置左右轮正反向系数
- `R0`：复位编码器计数、odometry 位姿，以及本轮定位/导航状态
- `D<actual_distance_m>`：根据本次实际测得位移给出新的编码器标定建议

## 3. Phase 3 进度

### 3.1 Phase 3A：建图可用的定位修正

已完成。

当前已经做到：

- `LocalizationTask` 输出 `predicted_pose / corrected_pose`
- `MappingTask` 已统一使用 `corrected_pose`
- `ODOM / EST / pred / corr` 调试链路已接通
- 静止状态下定位修正基本稳定

最近测试结论：

- 静止漂移 `<= 1 cell`
- 静止时 `mode=1`、`inliers` 和 `fit_mm` 基本正常

这说明：

**当前定位链路已经足以支撑“建图修正”，但还不适合直接接管控制闭环。**

### 3.2 Phase 3B：控制可用的融合定位

部分完成，未完全收口。

已经做过：

- `EST / CTRL` 双层位姿分离
- `CTRL = odometry + bounded slow correction` 的一版实现
- 避免 LiDAR 修正直接把 PID 闭环拉坏
- odometry 累计位移改为直接使用原始编码器速度，避免平滑速度导致里程缩水
- 平移积分改为使用中点航向，降低转弯同时前进时的积分偏差
- 相对位移控制改为按命令前进轴上的投影进度计算完成量，避免走弧线时用起点直线距离少算路程
- 基础运动控制的角度与距离参考统一回 `odometry`，避免 `control_pose` 慢速修正与里程进度混用
- `O` 调试输出补齐相对位移 `MOVE cmd/target/progress/remain`，并在动作结束后保留最近一次结果，便于人工复盘
- `O` 进一步补齐 `LMOVE valid/cmd/progress/dl/dr`，直接显示最近一次完成的相对位移段轮程，避免从累计 `ENC` 手工做差
- `L / R` 手动转向命令改为真正的相对转向，避免把绝对角度设成 `+/-10 deg` 带来额外姿态误差
- 调试口补齐 `ENC dl/dr/cntL/cntR/cal`、`R0` 和 `D...` 标定建议
- `D...` 标定建议现优先使用最近一次完成的 `P{x},{y}` 相对位移段轮程，避免累计轮程混入摆位或上一段动作
- 针对 `P1,0` 终点附近抖动，位置环在最后 `0.15 m` 不再强制 `MIN_BASE_SPEED`，并增加终端速度上限，从 `0.08 m/s` 收缩到约 `0.015 m/s`
- 倒车相对移动补齐 `-MAX_BASE_SPEED` 对称限幅，避免 `P-1,0` 反向速度输出过大时放大终点抖动
- 移动中的角度修正新增随 `base_speed` 收缩的限幅，避免终点附近角度环把一侧轮子拉成反向
- 角度控制死区从单一阈值改为 `1 deg / 2 deg` 迟滞，减少转角目标附近反复启停
- `O` 调试输出新增 `CTRLDBG pos_err/ang_err/turn/l_sp/r_sp/pwm`，用于区分位置环、角度环、电机输出或硬件噪声导致的抖动
- `O` 调试输出新增 `LCTRL`，保留最近一次相对位移动作结束前最后一轮驱动控制输出，便于动作结束后复盘终点抖动
- `O` 调试输出新增 `TCTRL`，保留最近一次原地转向有输出时的控制状态，便于动作结束后复盘转角抖动
- 新增 RAM-only PID 调参命令 `PK...`、host app PID 面板和 `car-pid-tuner` CLI；当前定位是辅助人工调参和后续 AI agent 接入，不自动接管控制
- 新增 `WSleft,right` 速度环隔离测试命令，可绕过位置环和角度环，直接给左右轮速度环设定值，用于判断前后抖动是否来自速度 PID、编码器测速或电机/供电链路
- 悬空速度环隔离测试确认原默认速度 PID `Kp=8000 Ki=33705 Kd=33` 在 `0.04 m/s` 附近会产生数千级 PWM 和 `0.28~0.46 m/s` 轮速尖峰；地面前进/转向候选参数已改为默认速度环 `Kp=1800 Ki=3000 Kd=0`
- 为继续调角度环和位置环，`O` 诊断输出新增 `PIDDBG / LIMDBG / MOTORDBG / ENCDBG`，可以直接观察 PID 内部分量、限幅状态、实际电机 PWM 与原始编码器测速量化

但实测也证明：

- LiDAR 抖动不能直接进入角度环
- 控制距离闭环当前仍主要依赖 `odometry`
- 地板材质变化、正反向不对称、轮胎打滑会明显影响位移精度
- 首轮实车量距已确认：前进 `P1.0,0.0` 时 `ODOM` 约 `0.984 m`、`MOVE progress` 约 `0.957 m`，实体仅约 `0.82 m`，说明当前主误差更接近轮比例/左右不对称，而不是相对位移进度统计本身
- 首轮记录里的倒车 `ENC dl/dr = -0.337 / 0.430 m` 是累计量；若扣除前进阶段累计值，倒车单段更接近 `dl/dr = -1.130 / -0.746 m`，当前更像反向左右轮不对称和比例偏差，而不是简单的符号翻转
- 手动 `L / R` 已恢复相对转向；转角卡顿/超调已加入第一轮控制稳定化修改，但尚未完成硬件复测
- 新版隔离测试已给出合并系数建议：`K0.990,1.069,0.776,1.308`
- 前进单段实体 `0.83 m`，`LMOVE dl/dr = 0.838 / 1.069 m`，对应 `left_err=1.0%`、`right_err=28.8%`
- 倒车单段实体 `-1.12 m`，`LMOVE dl/dr = -1.048 / -0.857 m`，对应 `left_err=-6.4%`、`right_err=-23.5%`
- 应用 `K0.990,1.069,0.776,1.308` 后，前进实体距离改善到 `0.97 m`，但 `CAL diag` 仍显示 `left_err=-9.0%`、`right_err=5.2%`、`asym=14.5%`
- 同一组系数下倒车本轮实体距离为 `-0.85 m`，与上一轮 `-1.12 m` 差异较大，说明单次 1 m 标定重复性不足
- 重新整理/固定线路后，即使默认 `K1.000,1.000,1.000,1.000`，前进结果也改善为实体 `0.88 m`、`LMOVE dl/dr = 0.919 / 0.982 m`，倒车结果改善为实体 `-0.96 m`、`LMOVE dl/dr = -0.968 / -0.932 m`
- 这表明此前的大幅波动很可能混入线束接触、编码器信号干扰、供电/地线或线束运动问题，不能只靠软件标定解释
- 重新理线后，`K0.957,0.992,0.896,1.030` 已通过连续 `3` 次前后 `1 m` 往返测试，最终与起点误差约 `5 cm`
- 尚未完成“刷入新版标定窗口后，带轻微航向修正的 `P1.0,0.0` 是否仍会系统性超走”的复测
- `WS` 悬空速度环隔离已完成初步复测；尚未完成落地负载下 `WS-0.04,-0.04`、`P-0.30,0` 和 `P-1.00,0` 的复测
- 最新地面调参显示转向慢和前进加速慢不能只靠统一提高速度环 `Kp` 解决；下一轮应基于新增诊断判断角度环限幅、位置环 base speed 和低速编码器量化的贡献
- `A90` 诊断确认角度 PID 输出长期被 `TURN_OUTPUT_LIMIT_INPLACE=0.08 m/s` 限制，已将原地转向上限提高到 `0.12 m/s`，等待重新烧录后复测
- 在 `rollback/before-deadzone-debug` 分支继续三环 PID 实测：默认速度环 `1800/3000/0` 在 `WS0.04,0.04` 下 1 s 后仍几乎未起步；`3500/6000/0` 能改善低速启动；`4500/9000/0` 会在右轮低速测速尖峰后反向修正，不适合作为默认值。角度环 `0.008/0/0.0005` 在 `A20` 样本中比 `0.006/0/0.0005` 更快且未见明显过冲。位置环 `0.8/0/0` 暂不继续加大，因为 `P0.30` 的 5 cm 终点误差主要来自 `POSITION_REACHED_THRESHOLD`，`P0.50` 又暴露右轮速度反馈反向尖峰，继续提高位置环会放大底层速度反馈问题

当前真实判断是：

**Phase 3B 还不能宣告完成。**

它现在的状态更接近：

- 架构方向已经明确
- 关键风险已经暴露
- 但控制融合仍需保守推进

## 4. Phase 4A 进度

### 4.1 已启动内容

`Phase 4A` 已启动，并已有第一版“已知目标点基础导航”。

当前新增能力：

- 地图坐标转换：`world <-> cell`
- 基础路径规划接口：`MappingTask_PlanPath(...)`
- 导航状态机：`NavigationTask`
- 分段执行：路径拆成短段，调用 `Start_Relative_Move(...)`

当前导航命令：

- `Jx,y`：导航到地图格点 `(x, y)`
- `J`：查看导航状态
- `C`：取消当前导航

当前导航实现特征：

- 这是**已知目标格点导航**
- 当前是**最小可用实现**
- 路径规划器是**轻量栅格规划器**
- 不是完整的 frontier exploration
- 不是出口搜索
- 不是 return-to-start 状态机

### 4.2 当前限制

当前导航仍有明显边界：

- 目标必须落在已知可通行区域
- 依赖当前地图和位姿估计质量
- 缺少成熟的失败恢复和重规划策略
- 尚未把路径绘制到 ASCII 地图
- 未实现探索/回程闭环

## 5. 当前实测状态

### 5.1 控制侧

当前控制已经回到“稳定优先”的状态：

- 手动控制恢复稳定
- 速度和转向上限已下调，适合建图测试
- 正反向编码器系数已独立可调
- 已可在串口侧直接查看累计轮程、计数值，并基于实测距离生成新的标定建议

但仍存在的问题：

- 当前里程计仍可能出现约 `10% ~ 20%` 的尺度误差
- 前进方向当前主要表现为右轮前进里程记大，建议把 `rf` 从 `1.000` 调到 `0.776`
- 后退方向当前主要表现为两侧倒车里程都记小，建议把 `lr` 调到 `1.069`、`rr` 调到 `1.308`
- 但应用合并系数后的单次结果出现波动，后续不应继续按单次 `D...` 建议大幅改系数，而应固定当前 `K` 做多轮重复取中位数
- 重新理线后默认系数已经显著改善，当前优先级应转为固定硬件线束和编码器信号，再建立默认系数重复性基线
- 当前推荐默认编码器系数为 `K0.957,0.992,0.896,1.030`
- 前后运动存在不对称
- 不同地板材质会明显改变结果
- 单次前进/后退后，物理位置仍可能偏移 `10 ~ 15 cm`
- 相对位移控制的剩余误差需要区分“纯尺度偏差”与“航向修正导致的路径进度计算偏差”
- 终点和转向抖动的第一轮软件缓解已完成，下一轮必须通过 `CTRLDBG` 实车记录确认是否仍存在左右轮目标速度一正一负或 PWM 正负翻转

这说明：

**当前系统不能依赖 dead reckoning 达到高精度往返回零。**

### 5.2 建图侧

建图当前状态可描述为：

- 可读
- 可用于定位和导航开发
- 但不属于高精度地图

典型现象：

- 静止时漂移小
- 运动时地图结构能出来
- 墙体仍可能偏厚
- 地图尺度和运动尺度受 odometry 影响较大

因此当前更适合的目标不是“厘米级精确回原点”，而是：

- 地图整体不崩
- 机器人在图中位置基本可信
- 能支撑基础导航和后续重规划

## 6. 当前资源与风险

最近一次构建后，资源占用大致为：

- RAM：`98.85%`
- FLASH：`22.06%`

当前最重要的工程风险是：

### 6.1 RAM 余量过小

导航模块加入后，RAM 已接近上限。

这意味着：

- 后续再增加大数组要非常谨慎
- 规划器、地图缓存、状态机都需要控内存
- 后续 frontier / return-to-start 前必须先做一轮资源收口

### 6.2 导航健壮性不足

虽然 `Phase 4A` 已开始，但当前导航还缺：

- 更稳健的重规划
- 失败恢复
- 更强的局部避障
- 更成熟的到达判定

### 6.3 运动尺度误差仍会影响地图与导航

当前实际表现说明：

- 地图质量仍受运动质量主导
- 速度、转弯速度、地板材质都会显著影响结果

## 7. 下一阶段建议

### Priority 1

收口 `Phase 3B`，但保持保守策略：

- 控制继续以稳定为优先
- 不把 LiDAR 修正硬接入 PID
- 继续保留 `odometry / control_pose / corrected_pose` 三层分工
- 在长距离导航验证前，先完成至少一轮前进/后退量距标定，并记录更新后的 `K...` 系数
- 继续完成 `P1,0 / P-1,0 / A90 / A-90` 抖动复测，优先观察 `CTRLDBG`，再决定是否继续调低角度 Kp、转向限幅或电机死区补偿

### Priority 2

继续推进 `Phase 4A` 最小导航闭环：

- 导航路径可视化
- 到达容差
- 失败重规划
- 更清晰的导航状态输出

### Priority 3

做一轮资源收口：

- 压缩导航/地图额外缓存
- 评估路径规划器内存占用
- 为后续 frontier / return-to-start 留出空间

### Priority 4

在 `Phase 4A` 稳定后，再进入：

- frontier exploration
- exit detection
- return-to-start

## 8. 当前阶段性结论

现在的项目状态可以概括为：

**“控制已稳定，建图已可用，定位修正已接入，实时地图已经可看，基础导航已经起步，但建图精度、运动尺度和导航健壮性仍需继续收口。”**

这个状态已经足以进入下一阶段导航开发，但开发策略必须继续保持：

- 分层
- 保守
- 可测
- 可回退

补充判断：

- 里程计尺度问题已经进入“有工具可测、有命令可调”的阶段
- 里程计尺度误差已从早期 `10% ~ 20%` 收敛到三轮往返后约 `5 cm` 起点误差；后续重点是确认线束固定、转向抖动和导航动作中的稳定性
- 终点附近抖动的当前首要软件嫌疑是“最低速度钳制 + 角度修正过强”，固件已加入保守限制，状态为等待硬件验证
