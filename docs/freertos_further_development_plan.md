# FreeRTOS 后续开发计划

## 文档状态

- 版本：`v1.2`
- 更新日期：`2026-05-12`
- 适用范围：当前 STM32F446 + FreeRTOS + LiDAR SLAM 固件

---

## 1. 当前系统状态

当前系统已经不是“RTOS 改造计划”阶段，而是“RTOS 上的功能系统集成阶段”。

现有能力包括：

- `FreeRTOS` 多任务调度
- 编码器 + IMU odometry
- LiDAR scan 解析
- 轻量 `LocalizationTask`
- 占据栅格建图
- 实时 ASCII 地图输出
- 已知目标格点导航初版

当前主任务链路：

1. `ControlTask`
2. `LiDARParseTask`
3. `LocalizationTask`
4. `MappingTask`
5. `CommTask`
6. `SafetyTask`
7. `NavigationTask`（逻辑服务型模块，已接入调度链）

---

## 2. 当前架构分层

### 2.1 状态层

系统当前已经实际形成三层位姿概念：

- `odom_pose`
  - 高频
  - 平滑
  - 控制距离闭环当前主要依赖它

- `control_pose`
  - 面向控制层的融合位姿
  - 当前仍保持保守使用

- `corrected_pose / EST`
  - 面向建图与导航
  - 来自 `LocalizationTask`

### 2.2 功能层

- **控制层**
  - 以稳定性优先
  - 当前不允许 LiDAR 修正直接破坏 PID 闭环

- **建图层**
  - 统一使用 `corrected_pose`

- **导航层**
  - 当前使用地图格点和估计位姿
  - 已具备最小路径执行能力

## 4. 当前进行中的阶段

### Phase 3B：控制可用的融合定位

当前状态：**进行中**

目标不是“把 LiDAR 直接喂给 PID”，而是：

- 保持控制稳定
- 让 LiDAR 修正逐步进入更高层状态估计
- 为导航提供更可信的位置参考

当前已知约束：

- LiDAR 修正有低频和抖动
- 不能直接进入角度闭环
- 控制距离闭环仍主要看 `odometry`
- 地板材质和正反向不对称会显著影响结果

当前结论：

**Phase 3B 仍需继续收口，但不再是阻止 Phase 4A 启动的绝对前置条件。**

### Phase 4A：已知目标点基础导航

当前状态：**已启动**

已经具备：

- 地图格点目标输入
- 基础路径规划接口
- 导航状态机
- 分段执行路径

当前命令：

- `Jx,y`：导航到目标格点
- `J`：查看导航状态
- `C`：取消导航

当前定位：

- 这是已知目标格点导航
- 不是完整自主探索
- 不是 return-to-start

## 5. 当前命令与调试接口

### 5.1 建图与地图查看

- `M / N`：开启/停止 LiDAR
- `G`：地图摘要
- `X1 / X2 / X4 ...`：导出 ASCII 地图
- `U1 / U0`：开启/关闭 `0.5 s` 实时缩略地图流

### 5.2 位姿与定位

- `O`：`ODOM / ENC / EST / CTRL / pred / corr / slow`
- `O` 同时包含 `MOVE cmd/target/progress/remain` 相对位移诊断信息，并保留最近一次动作结果直到下一次命令或复位
- `P`：定位状态摘要

### 5.3 编码器标定

- `K`：查看当前正反向标定系数
- `Klf,lr,rf,rr`：修改标定系数
- `R0`：复位编码器计数、odometry 位姿、LocalizationTask 和导航状态
- `D<actual_distance_m>`：基于当前累计轮程和实测位移给出新的标定建议

### 5.4 导航

- `Jx,y`：导航到地图格点
- `J`：查看导航状态
- `C`：取消导航

## 6. 当前主要问题

### 6.1 运动尺度误差

当前实测已经说明：

- 前进和后退存在不对称
- 地板材质变化会带来明显影响
- 单次往返后实际偏移仍可达 `10 ~ 15 cm`
- 当前里程计在部分工况下仍可能出现 `10% ~ 20%` 的尺度误差

最近一轮固件修正已经完成：

- odometry 位移累计直接使用原始编码器速度，避免平滑速度造成累计缩放
- 平移积分使用中点航向
- 相对位移控制按命令前进轴投影计算进度，避免浅弧行驶时把实际路程低估成起点直线位移
- 基础运动控制闭环统一使用 odometry 参考，不再把 `control_pose` 慢速修正混入 PID 主链路
- 串口调试补齐累计轮程、原始计数、复位和标定建议接口

但这些结论目前仍停留在桌面构建和代码审查层面，尚未完成硬件量距验证。

因此当前不应以“精准回零”作为导航前提。

### 6.2 建图精度仍有限

当前地图已经“可读”，但仍受以下因素影响：

- 运动速度过快
- 转弯速度过快
- odometry 尺度误差
- scan matching 仅为轻量实现

### 6.3 内存风险

当前资源占用大致为：

- RAM：`98.85%`
- FLASH：`22.06%`

这意味着后续新增模块必须优先考虑 RAM。

## 7. 后续开发优先级

### Priority 1：收口 Phase 4A

下一步优先把基础导航做成“可重复验证”的最小闭环：

- 路径可视化
- 到达容差
- 失败状态输出
- 重规划入口
- 更清晰的导航调试信息

### Priority 2：收口 Phase 3B

继续推进但保持保守：

- 不把 `corrected_pose` 直接送入 PID
- 继续维护分层位姿
- 逐步评估 `control_pose` 是否还有必要继续保留增强
- 优先完成 `R0 -> motion -> O -> D... -> K...` 的人工标定闭环，并把结果写入任务记录

### Priority 3：资源优化

重点检查：

- 地图规划缓存
- 导航路径缓存
- 运行时临时数组

目标是为后续 frontier / return-to-start 释放内存余量。

### Priority 4：进入完整导航状态机

在基础导航稳定后，再进入：

- frontier exploration
- exit detection
- return-to-start

---

## 8. 推荐开发顺序

推荐按以下顺序推进：

1. 稳定当前基础导航
2. 补导航路径显示
3. 加失败恢复和重规划
4. 做 RAM 收口
5. 再考虑 frontier
6. 最后做回程状态机

---

## 9. 当前阶段判断

当前系统已经具备“继续做导航”的条件，但前提是：

- 导航必须按“容错系统”来设计
- 不把 dead reckoning 当作高精度真值
- 允许分段执行、重定位、再规划

因此后续计划不需要推翻，但需要继续坚持：

- 分层
- 小步快跑
- 每一步都可测
- 遇到问题可快速回退
