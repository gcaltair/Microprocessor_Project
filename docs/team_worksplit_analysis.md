# 六人分工方案（按你原先“四人分工”同款格式）

> 目标：在你原有 4 个任务包基础上，扩展为 6 人协作方案，并补充一张「6人×每周×可交付物」排期表，方便直接执行与验收。

## 1：建图与定位核心算法实现（SLAM Core）

### 主要目标
实现嵌入式 SLAM 核心：在 STM32 上完成位姿估计修正 + 占用栅格地图构建，为规划器提供稳定输入。

### 具体工作内容
1. 占用栅格地图（OGM）
   - 设计轻量级 2D OGM 数据结构（静态内存分配）。
   - 根据迷宫尺寸（5×5，70cm/格）确定分辨率、地图范围、坐标原点与越界策略。
2. 扫描匹配
   - 实现轻量扫描匹配（ICP 或相关性匹配），用于修正里程计漂移。
   - 输出“匹配前/后位姿差异”供调参与验证。
3. 地图更新
   - 将校正后位姿 + LiDAR 点云进行光线投射更新。
   - 处理边界、遮挡、异常点、无效距离点。
4. 对外接口
   - 提供规划器可调用 API：`GetCurrentPose()`、`GetMap()`、`MapToWorld()/WorldToMap()`。

### 评估标准
**功能完整性（是否实现）**
- OGM 初始化正确（分辨率、尺寸、边界符合赛道）。
- 地图可持续更新（点云转栅格结果正确）。
- 扫描匹配可运行并可见修正效果。

**实现质量**
- 嵌入式运行帧率可接受，无明显卡顿。
- 传感器噪声下地图不发散，位姿不剧烈跳变。
- 全静态内存或受控内存策略，避免堆碎片。

**验证与操作**
- 提供串口/日志对比：匹配前后位姿误差。
- 输出地图轮廓（蓝牙调试助手/OLED/上位机可视化）。

## 2：导航与规划算法实现（Planning & Exploration）

### 主要目标
实现自主探索 + 路径规划，达成“入口→出口→入口”的自动导航闭环。

### 具体工作内容
1. 全局路径规划
   - 实现 A*，输入 OGM，输出可执行路径。
   - 增加路径平滑或拐点压缩，降低锯齿路径对控制环冲击。
2. 自主探索
   - 实现前沿检测（frontier detection）。
   - 设计前沿选择策略（距离/信息增益/可达性）。
3. 局部避障
   - 实现简化局部避障（势场法、窗口简化或重规划触发策略）。
4. 行为状态机
   - `探索模式` -> `导航至出口` -> `返回起点` -> `故障模式`。
   - 处理规划失败、超时、卡死重试。

### 评估标准
**功能完整性（是否实现）**
- A* 在已知地图稳定出路径。
- 自主探索可连续推进，无需人工指定路径点。
- 状态机模式切换正确，故障路径存在退出机制。

**实现质量**
- A* 运行时间不阻塞控制循环。
- 规划失败可自动重规划/重选目标。
- 输出路径经碰撞检查后再执行。

**验证与操作**
- 演示未知迷宫自主决策（非预设路线）。
- 多次完成“入口→出口”“出口→入口”基准流程。

## 3：实时操作系统（FreeRTOS）与系统架构集成

### 主要目标
把任务1/2/4/5/6 的功能统一到 FreeRTOS 架构，保证实时性、稳定性、可维护性。

### 具体工作内容
1. 接口统一与任务拆分
   - 设计任务边界：`SensorTask`、`MappingTask`、`PlanningTask`、`ControlTask`、`UITask`、`SafetyTask`。
2. 数据同步与通信
   - 使用队列/信号量/互斥锁管理跨任务数据。
   - 关键共享数据（位姿、地图索引、故障状态）加锁保护。
3. 集成联调
   - 整合各成员模块，建立统一初始化顺序与健康检查流程。
4. 调试优化
   - 测任务执行时间、堆栈占用、中断延迟，优化优先级和周期。

### 评估标准
**功能完整性（是否实现）**
- 各任务可并行稳定运行。
- 队列和锁机制正确使用，无裸全局变量竞争。
- DMA/中断链路正常（LiDAR、串口、编码器）。

**实现质量**
- 端到端延迟可测且在可接受范围。
- 连续运行 >10 分钟无死锁、无崩溃。
- 启动顺序清晰（外设就绪后再进入控制闭环）。

**验证与操作**
- 输出任务列表与堆栈/CPU占用信息。
- 任务异常可被 SafetyTask 检测并触发停车。

## 4：用户界面、安全机制与性能验证

### 主要目标
完成人机交互、安全机制、基准测试与演示资料支撑。

### 具体工作内容
1. 用户界面
   - OLED 显示：模式、位姿、电压、探索进度、目标点。
   - 按键防抖与功能绑定：开始/急停/回起点/模式切换。
   - ADC 电位器在线调参（速度上限/PID 参数）。
2. 安全机制
   - 看门狗、低电压检测、急停 PWM 关断、关键外设健康监测。
3. 性能验证与交付支持
   - 记录 Exit/Return 时间并计算性能分。
   - 协助演示视频脚本、README、提交清单一致性检查。

### 评估标准
**功能完整性（是否实现）**
- UI 信息完整，按键操作可靠。
- 蓝牙状态上报不阻塞主控制。
- 急停、低压、看门狗机制有效。

**实现质量**
- UI/遥测在独立任务中运行，不破坏实时控制。
- 文档结构清晰，可复现实验流程。

**验证与操作**
- 提供多次测试日志，展示一致性。
- 确保代码 ZIP 与视频演示行为一致。

## 5：现有代码在新硬件上调通（新增角色）

### 主要目标
将现有仓库代码完整迁移并跑通到新硬件平台，打通“可编译、可下载、可运行、可回归”的硬件闭环。

### 具体工作内容
1. 硬件抽象与引脚迁移
   - 核对新硬件引脚映射、外设实例、时钟树、定时器通道。
   - 更新 `*.ioc` 与底层驱动配置（UART/I2C/SPI/TIM/ADC/DMA）。
2. 驱动兼容性修复
   - 修复新旧板差异导致的初始化失败、DMA冲突、中断优先级问题。
   - 建立“最小可运行外设集”点亮流程（先串口、再电机、再传感器）。
3. 回归验证
   - 形成“每次合并后的硬件回归清单”：启动、传感器、控制、通信、地图更新、规划执行。
4. 风险兜底
   - 输出故障定位 SOP（上电失败、串口无输出、电机反转、LiDAR超时等）。

### 评估标准
**功能完整性（是否实现）**
- 主分支在新硬件可稳定启动和运行。
- 核心外设链路全部打通。

**实现质量**
- 回归流程可重复，问题可定位。
- 兼容性问题修复有记录，有版本追溯。

**验证与操作**
- 提供新硬件 bring-up 报告（映射表、问题单、修复记录）。
- 演示“冷启动到可控制”的完整链路。

### 主要目标

### 具体工作内容

   - 维护演示脚本与提交清单（代码 ZIP、技术文档、视频文件）。
3. 交付一致性审计
   - 校验“文档描述 = 视频演示 = 代码功能 = 实物行为”。

### 评估标准
**功能完整性（是否实现）**
- 实物装配可复现，接线可追溯。
- 文档齐全且结构清晰。

**实现质量**
- 文档更新及时，与代码版本同步。
- 提交材料完整，格式与命名规范统一。

**验证与操作**
- 提供交付前 checklist 并逐项签字确认。
- 提供至少一次“模拟提交”演练记录。

## 周会与验收建议（配套执行）

1. 每周固定两次联调：
   - 周中：接口回归（看是否破坏他人模块）
   - 周末：全链路演示（建图→规划→执行→返回）
2. 每次联调必须输出：
   - 可复现场景
   - 指标数据（周期、时延、成功率、耗时）
   - 问题清单 + Owner + 截止时间
3. 冻结策略：
   - 第5周末冻结功能，不再加新特性；
   - 第6周只做修复、验证、材料打包。
